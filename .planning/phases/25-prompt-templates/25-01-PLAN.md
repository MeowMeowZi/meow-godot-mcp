**8. tool_composition_guide (PROMPT-01)**
- Name: `"tool_composition_guide"`
- Description: `"Quick reference card mapping common game development tasks to MCP tool sequences with parameter examples"`
- Arguments: `[{"name": "task_category", "description": "Task category: scene_setup, ui_building, animation, scripting, debugging, testing (default: scene_setup)", "required": false}]`
- Lambda: Based on task_category argument, return a structured guide. Each category maps 3-5 common tasks to their tool sequences with concrete parameter JSON examples.
  - `scene_setup`: create_scene -> create_node -> set_node_property -> save_scene; instantiate_scene; find_nodes + batch_set_property
  - `ui_building`: create_node(Control) -> set_layout_preset -> create_stylebox -> set_theme_override -> set_container_layout; create_ui_panel for quick layouts
  - `animation`: create_animation -> add_animation_track -> set_keyframe -> set_animation_properties
  - `scripting`: write_script -> attach_script; read_script -> edit_script; connect_signal
  - `debugging`: get_scene_tree -> find_nodes; get_node_signals; get_game_output; eval_in_game; capture_game_viewport
  - `testing`: run_game -> get_game_bridge_status -> click_node -> get_game_node_property -> stop_game; run_test_sequence
  - Default (unknown category): return a summary card listing all 6 categories with 1-liner descriptions

**9. debug_game_crash (PROMPT-02)**
- Name: `"debug_game_crash"`
- Description: `"Systematic workflow for diagnosing game crashes and runtime errors using MCP diagnostic tools"`
- Arguments: `[{"name": "error_type", "description": "Error type: crash, null_reference, signal_error, script_error, scene_load_error (default: crash)", "required": false}]`
- Lambda: Based on error_type, return step-by-step diagnostic workflow:
  - `crash`: 1) get_game_output to read error logs, 2) read_script on the crashing script, 3) get_scene_tree to check scene structure, 4) find_nodes to verify expected nodes exist, 5) get_node_signals to check signal connections, 6) fix with edit_script, 7) run_game to re-test
  - `null_reference`: Focus on get_scene_tree + find_nodes to verify node paths, eval_in_game to test references
  - `signal_error`: Focus on get_node_signals + connect_signal/disconnect_signal
  - `script_error`: Focus on read_script + edit_script with line-level fixes
  - `scene_load_error`: Focus on list_project_files + open_scene + get_scene_tree
  - Each workflow should list exact tool names and typical parameter patterns

**10. debug_physics_issue (PROMPT-06)**
- Name: `"debug_physics_issue"`
- Description: `"Specialized workflow for diagnosing and fixing physics problems: collision failures, movement bugs, layer mismatches"`
- Arguments: `[{"name": "issue_type", "description": "Physics issue: no_collision, wrong_movement, falling_through, jitter, one_way_collision, tunneling (default: no_collision)", "required": false}]`
- Lambda: Based on issue_type, return targeted diagnostic steps using MCP tools:
  - `no_collision`: 1) find_nodes by type "CollisionShape*" to check shapes exist, 2) get_scene_tree to verify parent body types, 3) eval_in_game to read collision_layer/collision_mask, 4) batch_set_property to fix layers
  - `wrong_movement`: Check CharacterBody velocity, floor detection via eval_in_game, verify _physics_process in read_script
  - `falling_through`: Verify collision shapes assigned, check layer/mask compatibility, verify StaticBody for floors
  - `jitter`: Check _process vs _physics_process usage, verify delta usage in scripts
  - `one_way_collision`: Check one_way_collision property on CollisionShape
  - `tunneling`: Suggest CCD or increased physics ticks
  - Reference tools: find_nodes, get_scene_tree, eval_in_game, read_script, edit_script, batch_set_property, get_game_node_property, run_game

**11. fix_common_errors (PROMPT-08)**
- Name: `"fix_common_errors"`
- Description: `"Recovery guide for common MCP tool errors: node not found, scene not open, script syntax errors, property type mismatches"`
- Arguments: `[{"name": "error_pattern", "description": "Error pattern: node_not_found, no_scene_open, script_syntax, type_mismatch, game_not_running, permission_error (default: node_not_found)", "required": false}]`
- Lambda: Based on error_pattern, return recovery workflow:
  - `node_not_found`: 1) get_scene_tree to see actual tree, 2) find_nodes with partial name match, 3) check path format (use absolute from scene root)
  - `no_scene_open`: 1) list_open_scenes, 2) open_scene or create_scene, 3) retry original operation
  - `script_syntax`: 1) read_script to see current content, 2) edit_script to fix specific lines, 3) re-attach if needed
  - `type_mismatch`: Examples of correct property value formats (Vector2, Color, NodePath), use get_resource_info to check expected types
  - `game_not_running`: 1) run_game, 2) get_game_bridge_status to wait for connection, 3) retry runtime tool
  - `permission_error`: Check file paths with list_project_files, verify res:// prefix
  - Each entry should include the exact error message pattern and the tool sequence to recover

IMPORTANT: Every tool name referenced in prompt text MUST be from the 55-tool registry. Do NOT invent tool names. Do NOT reference tools like "get_collision_info" that don't exist.

The prompt text should be workflow-oriented -- numbered steps with tool names and parameter examples, not just descriptions.
