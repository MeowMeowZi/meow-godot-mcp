1. **MCPClient class**: Copy exactly from uat_phase9.py (connect, close, send, recv, initialize, call_tool).

2. **Game lifecycle helper**: Add a helper that:
   - Calls `run_game` with mode="main" to launch the game
   - Polls `get_game_bridge_status` in a loop (up to 10 seconds, 0.5s intervals) until `connected` is true
   - Returns success/failure for bridge connection

3. **Test list** (aim for ~13-15 tests covering all BRDG requirements):

**BRDG-01 (Bridge Connection):**
- Test 1: `test_tools_list_has_38_tools` -- tools/list returns 38 tools including inject_input, capture_game_viewport, get_game_bridge_status
- Test 2: `test_run_game_and_bridge_connects` -- run_game mode=main succeeds, then poll get_game_bridge_status until connected=true within 10 seconds
- Test 3: `test_bridge_status_shows_connected` -- After bridge connects, get_game_bridge_status returns {"connected": true, "session_id": >0}

**BRDG-02 (Key Injection):**
- Test 4: `test_inject_key_press` -- inject_input type=key, keycode="W", pressed=true returns success
- Test 5: `test_inject_key_release` -- inject_input type=key, keycode="W", pressed=false returns success

**BRDG-03 (Mouse Injection):**
- Test 6: `test_inject_mouse_click` -- inject_input type=mouse, mouse_action="click", position={x:100,y:100}, button="left", pressed=true returns success
- Test 7: `test_inject_mouse_move` -- inject_input type=mouse, mouse_action="move", position={x:200,y:200} returns success
- Test 8: `test_inject_mouse_scroll` -- inject_input type=mouse, mouse_action="scroll", position={x:100,y:100}, direction="up" returns success

**BRDG-04 (Action Injection):**
- Test 9: `test_inject_action_press` -- inject_input type=action, action_name="ui_accept", pressed=true returns success
- Test 10: `test_inject_action_release` -- inject_input type=action, action_name="ui_accept", pressed=false returns success

**BRDG-05 (Game Viewport Capture):**
- Test 11: `test_capture_game_viewport` -- capture_game_viewport returns ImageContent with type "image", mimeType "image/png", non-empty data
- Test 12: `test_capture_game_viewport_base64_is_valid_png` -- decode base64 data, check PNG signature (first 8 bytes: 137 80 78 71 13 10 26 10)
- Test 13: `test_capture_game_viewport_with_resize` -- capture_game_viewport with width=320 returns ImageContent (just verify success, size may vary)

**Error cases:**
- Test 14: `test_inject_input_missing_type` -- inject_input without type returns error
- Test 15: `test_stop_game_and_bridge_disconnects` -- stop_game succeeds, then get_game_bridge_status shows connected=false (final cleanup test)

4. **Test execution flow**:
   - Connect to MCP server, initialize
   - Run test 1 (tools/list)
   - Run test 2 (launch game + wait for bridge) -- this is the setup for all subsequent tests
   - Run tests 3-14 (all require game running)
   - Run test 15 (stop game -- must be last)
   - Print summary, exit with appropriate code

5. **Important patterns from research**:
   - capture_game_viewport returns result with data field (ImageContent), same parsing as uat_phase9.py for capture_viewport
   - For ImageContent: result["result"]["content"] is an array, first item has type "image", data, mimeType
   - For regular tool results: result["result"]["content"][0]["text"] contains JSON string
   - Timeout for capture_game_viewport should be higher (15 seconds) since it involves cross-process communication
   - The call_tool method in MCPClient may need an extended timeout for capture_game_viewport

6. **call_tool return handling**:
   Same pattern as uat_phase9.py: call_tool returns the raw result dict. For ImageContent responses, check content[0]["type"] == "image". For regular text responses, json.loads(content[0]["text"]).
