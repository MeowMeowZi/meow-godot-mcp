<objective>
Build the bridge executable that AI clients (Claude Desktop, Cursor) spawn as a subprocess. It reads JSON-RPC from stdin, relays to the GDExtension's TCP server, and writes responses back to stdout.

Purpose: The bridge solves the fundamental constraint that a GDExtension (shared library inside Godot) cannot be spawned as a subprocess by AI clients. The bridge is the MCP stdio transport endpoint.
Output: A standalone executable (~50KB) that can be configured in claude_desktop_config.json as an MCP server.
</objective>

TCP Protocol Contract (bridge <-> GDExtension):
- Transport: TCP localhost, default port 6800 (per user decision)
- Framing: Newline-delimited JSON (each message is one line terminated by \n)
- Message format: JSON-RPC 2.0 (same as MCP stdio transport)
- The bridge is a transparent relay: it does NOT parse, validate, or modify messages
- Bridge reads from stdin, sends to TCP. Reads from TCP, writes to stdout.

3. Create `bridge/tcp_client.h`:
   - Class `TcpClient`:
     - Constructor: `TcpClient(const std::string& host, int port)`
     - `bool connect_with_retry(int max_retries = 15, int retry_interval_ms = 2000)` -- fixed 2-second interval per user decision, no exponential backoff. Logs each attempt to stderr.
     - `bool send_message(const std::string& message)` -- sends message + "\n" over TCP. Returns false on error.
     - `bool receive_message(std::string& out_message)` -- reads until newline from TCP. Returns false on disconnect. Uses non-blocking or select() with timeout to avoid blocking forever.
     - `void disconnect()`
     - `bool is_connected() const`
   - Private members:
     - Platform socket handle (`SOCKET` on Windows, `int` on POSIX)
     - `std::string read_buffer` -- accumulates TCP data for newline framing
     - `std::string host`
     - `int port`
     - `bool connected`
   - Platform includes guarded by `#ifdef _WIN32` (winsock2.h) vs POSIX (sys/socket.h, etc.)

4. Create `bridge/tcp_client.cpp`:
   - Constructor: store host/port, initialize socket to invalid
   - `connect_with_retry`:
     - On Windows: call WSAStartup() once (use a static bool to track)
     - Create TCP socket (AF_INET, SOCK_STREAM)
     - Resolve host to sockaddr_in (localhost = 127.0.0.1)
     - Attempt connect(). On failure, log to stderr ("Attempt N/15: connection refused, retrying in 2s..."), sleep for retry_interval_ms (fixed 2 seconds, NO exponential backoff per user decision), retry.
     - On success, set connected=true, return true
     - On max retries exceeded (15 attempts = 30 seconds total), log error "Failed to connect after 15 attempts (30s timeout). Ensure Godot is running with MCP Meow plugin enabled.", return false
   - `send_message`:
     - Append "\n" to message
     - Send full buffer via send() in a loop (handle partial sends)
     - Return false on error (set connected=false)
   - `receive_message`:
     - Use select() with 100ms timeout to check readability (non-blocking check)
     - If readable, recv() into temporary buffer, append to read_buffer
     - Check read_buffer for newline. If found, extract first complete line into out_message, keep remainder in read_buffer
     - Return false on recv() error or disconnect (0 bytes)
     - Return true when a complete message is available, false (with no error) when timeout expires with no data
   - `disconnect`:
     - Close socket, set connected=false
     - On Windows: closesocket(). On POSIX: close().
   - IMPORTANT: All platform-specific code guarded by `#ifdef _WIN32` / `#else`
   - IMPORTANT: WSACleanup() in a static destructor or atexit handler on Windows

1. Create `bridge/main.cpp`:
   - Parse command-line arguments:
     - `--port <N>` -- TCP port (default 6800, per user decision)
     - `--host <addr>` -- TCP host (default "127.0.0.1")
     - `--help` -- print usage to stderr and exit
   - Simple arg parsing: loop argv, match "--port" -> next arg is port, etc. No library needed.
   - Main flow:
     ```
     a. Create StdioTransport
     b. Create TcpClient(host, port)
     c. Log startup info to stderr
     d. Connect to GDExtension with retry:
        - If connection fails after all retries, log error to stderr, exit(1)
     e. Main relay loop (runs until stdin EOF or TCP disconnect):
        - Check for incoming TCP message (non-blocking via select timeout):
          - If message received from TCP, write to stdout via StdioTransport
        - Check for incoming stdin message (use select/poll on stdin, or use a separate thread):
          - If message received from stdin, send to TCP via TcpClient
        - If TCP disconnects, attempt reconnection with fixed 2-second interval (indefinitely per user decision), log to stderr
        - During disconnected period, buffer stdin messages in queue
        - If stdin EOF, disconnect TCP and exit(0)
     f. Cleanup: disconnect TCP
     ```
   - THREADING APPROACH for the relay loop: Use two threads:
     - Thread 1 (stdin reader): Blocks on std::getline. When a line arrives, push to a thread-safe queue.
     - Main thread: Polls the stdin queue and TCP receive in a loop with a small sleep (10ms) to avoid busy-waiting.
     - This avoids the complexity of select() on stdin (which is difficult/impossible on Windows for console handles).
   - Thread-safe queue: Simple wrapper with `std::mutex` + `std::queue<std::string>` + `std::condition_variable` (or just mutex+queue with try_pop).
   - Graceful shutdown: When stdin thread gets EOF, set a `std::atomic<bool> running = false`. Main loop checks `running` and exits.
   - When TCP disconnects mid-session per user decision: log to stderr, attempt reconnect with fixed 2-second interval, indefinitely (bridge stays alive). During reconnect, buffer stdin messages in the queue (they will be sent when TCP reconnects). After reconnect, MCP handshake must be re-performed by the AI client (new session).
