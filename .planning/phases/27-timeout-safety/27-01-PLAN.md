<objective>
Add bounded wait times to the IO thread (30s) and game bridge deferred requests (15s), with stale response discard to prevent response corruption across requests.

Purpose: The IO thread's response_cv.wait() has no timeout -- if the main thread hangs or a deferred response never arrives, the MCP connection is effectively dead. 9 of 10 deferred game bridge operations also have no timeout. This plan adds timeouts to both paths and ensures late responses after timeout are safely discarded.

Output: Modified mcp_server.h/cpp and game_bridge.h/cpp with timeout support and stale response protection.
</objective>

**In src/mcp_server.cpp -- queue_deferred_response (around line 127):**

Add stale response detection at the top of the function, BEFORE the error enrichment logic. After line 128 (`nlohmann::json enriched_response = response;`), add:

```cpp
    // TIMEOUT-03: Discard stale responses from timed-out requests
    {
        std::lock_guard<std::recursive_mutex> lock(queue_mutex);
        if (io_pending_request_id.is_null()) {
            // No request being waited on -- IO thread already timed out and moved on
            return;
        }
        if (response.contains("id") && response["id"] != io_pending_request_id) {
            // Response ID does not match current pending request -- stale, discard
            return;
        }
    }
```

This ensures that: (a) if the IO thread already timed out (io_pending_request_id is null), the late response is discarded; (b) if the response is for a different request than what's being waited on, it's discarded.

**Key invariant:** `io_pending_request_id` is set in `process_message_io` (IO thread, under lock), read in `queue_deferred_response` (main thread, under lock), and cleared in `io_thread_func` after sending or timing out (IO thread, under lock). All access is protected by `queue_mutex`.
